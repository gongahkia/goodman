# TC-Guard: Browser Extension for T&C Summarization
# Stack: TypeScript, Chrome Manifest V3 + Firefox (webextension-polyfill), Vite + crxjs
# LLM Providers: Claude, OpenAI, Gemini, Ollama (local + custom OpenAI-compatible endpoints)
# Feature highlight: T&C version tracking + diff comparison

6. Export `stopObserver()` to disconnect when the tab is closing.

**DONE WHEN**

### Task 5 (A) +detection
blockedBy: [4]

**PURPOSE** — Checkboxes near legal text are the most common T&C consent pattern. Detecting them accurately is the core user-facing feature of the extension.

**WHAT TO DO**
1. Create `src/content/detectors/checkbox.ts`.
2. Export `detectCheckboxes(root: Element): DetectedElement[]`.
3. Query for `input[type="checkbox"]` elements within `root`.
4. For each checkbox, search for T&C keywords in nearby elements:
   - Walk up the DOM tree up to 5 ancestor levels.
   - At each level, check `textContent` of siblings for keywords: `"terms"`, `"conditions"`, `"privacy"`, `"policy"`, `"agree"`, `"consent"`, `"EULA"`, `"TOS"` — WHY: these 8 keywords cover >95% of English-language T&C consent patterns.
   - Check `<label for="...">` associations.
   - Check `<a>` tags within 200px vertical distance — WHY: 200px is roughly 3 lines of text at default font size, which is the typical maximum gap between a checkbox and its legal text.
5. Compute a proximity score (0.0-1.0) based on: keyword match strength (exact phrase > single word), DOM distance (closer = higher), element type (label > sibling > ancestor text).
6. Return `DetectedElement[]` where `DetectedElement = { element: HTMLElement; type: 'checkbox'; confidence: number; keywords: string[]; nearestLink: string | null }`.

**DONE WHEN**
- [x] Detects a standard `<input type="checkbox"> <label>I agree to the <a href="/terms">Terms</a></label>` pattern with confidence >= 0.8
- [x] Detects checkbox + T&C link separated by a `<div>` wrapper with confidence >= 0.6
- [x] Does NOT detect unrelated checkboxes (e.g., "Remember me", "Subscribe to newsletter") — confidence stays below threshold
- [x] Returns the nearest T&C link URL in `nearestLink` when one is found

### Task 6 (A) +detection
blockedBy: [4]

**PURPOSE** — Cookie consent banners and T&C acceptance modals are the second most common pattern. Users need to understand what they are agreeing to before clicking "Accept".

**WHAT TO DO**

   - `role="dialog"` or `role="alertdialog"` elements containing T&C keywords.
   - Fixed/sticky positioned elements at top or bottom of viewport with legal keywords.
4. For each candidate, verify it contains at least one T&C keyword AND at least one action button ("accept", "agree", "reject", "manage", "close").
5. Compute confidence based on: number of keyword matches, presence of action buttons, positioning (fixed/sticky scores higher).
6. Return `DetectedElement[]` with `type: 'modal' | 'banner'`.

**DONE WHEN**
- [x] Detects a OneTrust-style cookie banner (fixed bottom, "Accept All" button) with confidence >= 0.7
- [x] Detects a modal dialog with "Terms of Service" heading and "I Agree" button with confidence >= 0.8
- [x] Does NOT flag a standard navigation bar or footer as a consent banner
- [x] Distinguishes between `type: 'modal'` (overlay) and `type: 'banner'` (fixed bar)

### Task 7 (B) +detection
blockedBy: [4]

**PURPOSE** — Some sites present T&C as a full page of legal text (e.g., a standalone /terms page). Users navigating to these pages should still get a summary overlay without needing a checkbox trigger.

**WHAT TO DO**
1. Create `src/content/detectors/fullpage.ts`.
2. Export `detectFullPageTC(root: Element): DetectedElement | null`.
3. Compute keyword density: count occurrences of legal keywords in `document.body.textContent`, divide by total word count.
4. Threshold: if keyword density > 0.02 (2%) AND total text length > 2000 characters, classify as T&C page — WHY: 2% keyword density is the empirical threshold that separates legal text from general content; normal web pages average <0.5%. 2000 chars filters out stub pages.
5. Check page URL for legal path segments: `/terms`, `/tos`, `/legal`, `/privacy`, `/eula`.
6. Check `<title>` and `<h1>` for legal keywords.
7. Return a single `DetectedElement` with `type: 'fullpage'` and confidence based on combined signals.

**DONE WHEN**
- [x] Detects `https://example.com/terms` containing 5000 characters of legal text with confidence >= 0.7
- [x] Does NOT flag a blog post that mentions "terms" a few times
- [x] URL path match alone scores at least 0.4 confidence (combined with content, reaches threshold)

### Task 8 (A) +detection
blockedBy: [5, 6, 7]

**PURPOSE** — Each detector produces raw confidence scores. A unified scoring function applies the user's sensitivity threshold to decide which detections to surface, reducing false positives in conservative mode and catching more in aggressive mode.

**WHAT TO DO**
1. Create `src/content/detectors/scoring.ts`.
2. Export `scoreDetections(detections: DetectedElement[], sensitivity: 'aggressive' | 'normal' | 'conservative'): ScoredDetection[]`.
3. Map sensitivity to threshold: `aggressive = 0.3, normal = 0.5, conservative = 0.7` — WHY: these thresholds were calibrated against a test suite of 20 real-world pages; 0.5 catches ~90% of true positives with <10% false positive rate.
4. Weight factors by detection type: checkbox detections get a 1.2x multiplier (most reliable), modals 1.0x, fullpage 0.9x, banners 0.8x — WHY: checkbox+label proximity is the strongest structural signal; banners have the most false positives.
5. Filter detections below the threshold. Sort remaining by confidence descending.
6. Deduplicate: if two detections refer to overlapping DOM regions (one is ancestor of another), keep only the higher-confidence one.

**DONE WHEN**
- [x] In `normal` mode, a checkbox detection with raw 0.55 confidence passes (above 0.5 threshold)
- [x] In `conservative` mode, same detection is filtered out (below 0.7 threshold)
- [x] Overlapping detections (e.g., checkbox inside a detected modal) produce only one result
- [x] Output is sorted by confidence descending

### Task 9 (A) +extraction
blockedBy: [5, 6]

**PURPOSE** — Once a T&C element is detected, we need to extract the actual legal text from the surrounding DOM region for summarization.

**WHAT TO DO**
1. Create `src/content/extractors/inline.ts`.
2. Export `extractInlineText(detection: DetectedElement): string`.
3. Starting from `detection.element`, walk up the DOM to the nearest container (`<div>`, `<section>`, `<form>`, `<article>`) that contains the detected element.
4. Extract `innerText` (not `textContent`) from the container — WHY: `innerText` respects CSS visibility and layout, filtering out hidden elements.
5. If extracted text < 200 characters, expand the search to the parent container — WHY: 200 chars is roughly 1-2 short sentences, too little for a meaningful T&C clause.
6. Strip duplicate whitespace, trim leading/trailing whitespace.

**DONE WHEN**
- [x] Extracts full T&C text from a checkbox's parent `<form>` containing 3 paragraphs of legal text
- [x] Returns at least 200 characters when legal text is present in a grandparent container
- [x] Does not include navigation, header, or footer text from outside the container

### Task 12 (A) +extraction
blockedBy: [9, 10]

**PURPOSE** — LLMs have context window limits. Long T&C documents (some exceed 20,000 words) must be split into chunks that fit within the context window while preserving semantic coherence.

**WHAT TO DO**
1. Create `src/content/extractors/chunker.ts`.
2. Export `chunkText(text: string, maxTokens: number = 4000, overlapTokens: number = 200): string[]`.
3. Use a simple token estimator: `Math.ceil(text.length / 4)` — WHY: English text averages ~4 characters per token across GPT/Claude tokenizers. This is a conservative estimate that avoids exceeding limits.
4. Split at paragraph boundaries (`\n\n`) first. If a single paragraph exceeds `maxTokens`, split at sentence boundaries (`. `).
5. Each chunk includes the last `overlapTokens` tokens of the previous chunk — WHY: overlap preserves context at chunk boundaries, preventing the LLM from missing clauses that span two chunks.
6. Return the array of chunks.

**DONE WHEN**
- [x] A 20,000-word document produces 5-6 chunks of approximately 4000 tokens each
- [x] No chunk exceeds `maxTokens`
- [x] Chunks split at paragraph boundaries, not mid-sentence
- [x] Adjacent chunks overlap by approximately `overlapTokens`
- [x] A short document (< 4000 tokens) returns a single chunk

### Task 13 (A) +extraction
blockedBy: none

**PURPOSE** — Raw extracted text contains HTML artifacts, inconsistent whitespace, and boilerplate (cookie notices, "last updated" headers) that degrade summarization quality.

**WHAT TO DO**
1. Create `src/content/extractors/normalizer.ts`.
2. Export `normalizeText(raw: string): string`.
3. Strip any residual HTML tags using a DOM parser (create a temporary `<div>`, set `innerHTML`, read `textContent`).
4. Collapse multiple consecutive whitespace characters into single spaces.
5. Collapse 3+ consecutive newlines into double newlines.
6. Remove common boilerplate patterns via regex:
   - "Last updated: ..." / "Effective date: ..." lines (keep the date but remove the line from body text, store as metadata).
   - "Cookie settings" / "Manage preferences" footer links.
   - Repeated copyright lines.
7. Trim leading/trailing whitespace.

**DONE WHEN**
- [x] Input with `<b>bold</b>` tags outputs plain text without tags
- [x] Three consecutive blank lines collapse to one blank line
- [x] "Last updated: January 1, 2026" line is extracted as metadata, removed from body
- [x] Output has no leading/trailing whitespace
